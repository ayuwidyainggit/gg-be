# Plan — SX-2361 / SX-2364 Latest Location Pipeline (Mobile BE + Web BE)

> Mode: maintenance-stability (bugfix + small cross-service feature) dengan slice planning yang eksplisit karena target service behavior dan integrasi eksternal.

## Goal
Memastikan icon user di halaman **Live Monitoring Distributor** bergerak mengikuti koordinat device terbaru (bukan planned outlet). Hasil utama slice:
1. Initial load distributor map mengembalikan **latest known actual location** per salesman dengan fallback eksplisit ke `null` saat kosong.
2. Pipeline realtime dari Mobile BE ke FE Web (event `location.updated` / `location_updated`) tersedia end-to-end minimal: publisher Mobile BE → exchange RabbitMQ → queue `web-monitoring-location` → consumer `pjp` (Web BE) → push WebSocket ke FE.
3. Definisi "latest" eksplisit dan teruji di code.

## Non-goals
- Tidak menambah endpoint monitoring lain di luar `live-monitoring-distributor` dan `monitoring_locations/details` serta plumbing realtime minimal.
- Tidak me-rewrite controller, service, atau repository monitoring yang tidak terdampak.
- Tidak menambah dependency baru di luar yang sudah ada di repo (pola `streadway/amqp` di `sales`/`master`, pola `gorilla/websocket` belum ada; lihat Decision).
- Tidak mengubah kontrak `mobile.visits` atau `mobile.attendances`; publisher akan menjadi path baru atau menempel pada flow existing yang jelas.

## Scope
- **Service target utama:** `pjp` (Web BE yang punya endpoint distributor map) dan `mobile` (BE mobile yang punya akses GPS).
- **Service opsional/terkait:** `sales` atau `master` jika dipakai sebagai referensi pola RabbitMQ wrapper.
- **Database/Schema:** tidak membuat tabel baru di luar yang benar-benar dibutuhkan; jika belum ada tabel `mobile.current_location`, gunakan tabel existing (misal `mobile.attendances`/`mobile.visits`) atau tambah hanya dengan justifikasi dan migration mengikuti pola repo.
- **Topology:** exchange `location.exchange` (topic/direct sesuai pola repo), queue `web-monitoring-location`, binding `location.updated`, DLQ minimal sesuai pola repo.

## Requirements
1. Initial load `GET /scylla-pjp/v1/live-monitoring-distributor` mengembalikan field latest actual location per salesman.
2. Field harus eksplisit: `current_longitude`, `current_latitude`, `current_coordinate_at`, `current_coordinate_source` (sudah ada) atau nama flat `latest_*` jika FE SX-2363 sudah memutuskan konvensi; kontrak final diputuskan saat eksekusi task A3.
3. Definisi source priority eksplisit di code dan test:
   - `mobile.attendances` (type=2) terbaru → `attendance_checkout`
   - `mobile.attendances` (semua type) terbaru → `attendance`
   - `mobile.visits` terbaru → `mobile_visit`
   - `pjp.outlet_visit_list` terbaru → `outlet_visit_list`
   - lalu: `null` (jangan fallback ke planned).
4. Service distributor tidak boleh menghapus current coordinate hanya karena salesman belum clock-in pada tanggal request.
5. Mobile BE mem-publish event ke RabbitMQ saat koordinat device baru tersedia. Payload minimum:
   ```json
   { "event":"location.updated", "emp_id": 1001, "latitude":-6.21, "longitude":106.82, "recorded_at":"2026-06-29T10:20:00Z" }
   ```
6. Validasi payload: `emp_id` valid (auth user), latitude `[-90,90]`, longitude `[-180,180]`, `recorded_at` ISO-8601 UTC, anti-zero/null.
7. Topology: `location.exchange` → routing key `location.updated` → queue `web-monitoring-location`, dengan DLQ sesuai pola repo.
8. `pjp` (Web BE) consumer `web-monitoring-location` mengirim event WebSocket `location_updated` ke FE (payload minimum: `emp_id`, `latitude`, `longitude`, `recorded_at`).
9. Consumer idempotent: jika FE sudah menerima event dengan `recorded_at` lebih baru, abaikan yang lebih lama.
10. Push harus sampai ke FE klien yang sedang menampilkan distributor map; jika arsitektur FE pakai room distributor, broadcast ke room berisi emp tsb.
11. Update `monitoring_locations/details` (SX-2097 sudah selaras) hanya jika mempengaruhi `current_*`; jangan ubah kontrak section lain.
12. Tidak menambah dependency baru.

## Acceptance Criteria
- Initial load distributor map mengembalikan `current_*` non-zero/non-null untuk salesman yang punya actual location pada hari itu, termasuk jika belum clock-in.
- Bila tidak ada data actual sama sekali, response tetap `current_longitude: 0` atau `null` sesuai style repo, dan `current_coordinate_at: null`; tidak fallback ke `route_outlet_history` longitude/latitude.
- Source field `current_coordinate_source` salah satu dari: `attendance_checkout`, `attendance`, `mobile_visit`, `outlet_visit_list`, atau string kosong.
- Publish event `location.updated` dari Mobile BE muncul di exchange `location.exchange` dengan routing key benar, dan persistent.
- Queue `web-monitoring-location` menerima event dan ack/nack benar; DLQ aktif sesuai pola repo.
- Consumer `pjp` melakukan push WebSocket `location_updated`; event sampai ke FE klien uji.
- Field `current_coordinate_at` berupa epoch detik, dan TZ konsisten (UTC untuk `recorded_at`, hari bisnis dikonversi di service `pjp`).
- Test service + repository lulus: `rtk go test ./service/live_monitoring/...` dan `rtk go test ./repository/live_monitoring/...`.
- Diff minimum dan tidak menyentuh endpoint monitoring di luar scope.

## Constraints
- Layering: Controller → Service → Repository → DB. Tambah controller/service/repo bila butuh, jangan lewat langsung.
- Service `pjp` baca DB untuk `mobile.*` langsung (pola existing di `get_distributor_repository.go`) atau via view, bukan panggil HTTP ke `mobile` BE.
- Mobile BE publish event ke broker; tidak publish langsung ke `pjp` HTTP.
- Repo policy: shell workflow prefix `rtk`; tidak commit `.env` atau secret tracked.
- Jangan copy credential/JWT/cookie dari Jira/Issue ke source, test fixture, log, commit, atau artifact.
- Gunakan pola `pjp/database/migrate/` (`*.up.sql` / `*.down.sql`) untuk migration jika memang perlu tabel baru.

## Risks
- **Definisi "Web BE" user vs repo lokal**: kontrak user menyebut Mobile BE + Web BE, sedangkan repo lokal punya `pjp` sebagai service distributor. Plan ini mengasumsikan `pjp` adalah Web BE (punya endpoint FE baca). Bila ternyata beda, worklist perlu reassign owner.
- **Belum ada wrapper RabbitMQ di `pjp`**: dependensi `streadway/amqp` belum di-`go.mod` `pjp`; plan butuh keputusan apakah reuse sales wrapper (tidak boleh, beda module) atau tambah dependency lokal. Plan men-defer keputusan ini ke Task A1.
- **Belum ada library WebSocket di `pjp`**: kemungkinan perlu `gorilla/websocket`. Plan Task A4 men-defer pilihan; default rekomendasi: tambah dependency minimal, atau polling fallback bila FE belum siap.
- **Mobile FE tidak mengirim `/location`**: kontrak publisher butuh kajian apakah FE siap; task publisher di BE Mobile harus siap menerima payload jika FE mulai kirim.
- **Toleransi terhadap existing field naming FE**: FE SX-2363 mungkin sudah memutuskan nama field; koordinasi dengan `current_*` lama wajib.
- **Validasi tenant/`:cust_id` filter**: source `mobile.*` sudah punya filter custID; plan harus verifikasi ini tidak bocor.

## Decisions/Assumptions
- **Owner Web BE = `pjp`**: berdasarkan bukti endpoint distributor map berada di `pjp`. Asumsi: prompt "Web BE" di kontrak user merujuk ke `pjp`. Jika ternyata berbeda, plan perlu restrukturisasi.
- **Owner Mobile BE = `mobile`**: berdasarkan bukti repo mobile service.
- **Contract response existing `current_*` dipakai ulang dulu**: daripada menambah field `latest_*` yang bisa tabrakan dengan FE, prioritaskan field existing. Bila FE membutuhkan nama `latest_*`, mapping di service.
- **Definisi "latest" memakai timestamp `created_at` server (mobile.visits/attendances) karena `recorded_at` user belum masuk ke schema**: ini adalah keterbatasan pipeline jika belum ada publisher. Setelah publisher jalan, idealnya pakai `recorded_at` di response. (Asumsi eksplisit, dan terbatas pada slice ini.)
- **Publisher prioritas reuse tabel `mobile.visits`**: bila tidak ada tabel `current_location`, Mobile BE publisher cukup insert ke `mobile.visits` (atau temp table) lewat flow existing. Tabel dedicated `mobile.current_location` baru dibuat jika FE/BE butuh view spesifik. (Asumsi yang wajib diverifikasi Task A2.)
- **Format `recorded_at` ISO-8601 UTC** di payload event agar konsisten dengan service `pjp` yang memakai `Asia/Jakarta` untuk business day; konversi dilakukan di service `pjp`.
- **Initial-load bugfix yang paling minimum dan tertinggi dampak** adalah melepas gate `hasAttendanceOnRequestedDate` di `enrichDistributorMonitoringData` untuk current coordinate, **dengan tetap mempertahankan reset untuk field attendance/clock_out** agar tidak memberi kesan salesman clock-in padahal tidak.
- **TDD required**: production logic repository + service berubah. Existing `sqlmock` test pattern bisa dipakai.

## Non-negotiable Implementation Invariants
- **INV-1.** Worker (lane implementasi) **tidak boleh menghapus planned route coordinate** (`longitude`/`latitude` di `LiveMonitoringDestinationData`) yang dipakai marker outlet; hanya boleh menambah atau memisahkan field untuk marker user.
- **INV-2.** Worker **tidak boleh menambahkan fallback ke planned coordinate** ketika actual location kosong. Marker user boleh hilang (value 0/null) tapi outlet marker tetap.
- **INV-3.** Worker **tidak boleh mengedit endpoint monitoring lain** di luar `live-monitoring-distributor` dan `monitoring_locations/details` (dan hanya jika ada efek langsung ke `current_*`).
- **INV-4.** Worker **tidak boleh menambah dependency baru** di luar pilihan yang sudah ada di `pjp/go.mod` atau `mobile/go.mod` (pengecualian: dependency RabbitMQ/WebSocket boleh ditambah **hanya** setelah ada approval user karena termasuk arsitektur baru). Default: tetap tanpa dependency baru.
- **INV-5.** Worker **harus** menambahkan test minimal untuk definisi "latest" (priority + validasi) sebelum mengubah repository.
- **INV-6.** Worker **harus** menjaga invariant layering: tidak ada direct repository call dari controller.
- **INV-7.** Service writer/publisher event `location.updated` **harus** berisi field `emp_id` (numeric sesuai domain), `latitude`/`longitude` valid, dan `recorded_at` ISO-8601 UTC. Tidak ada field kosong.
- **INV-8.** Consumer `pjp` **harus** idempotent: payload dengan `recorded_at` yang lebih lama dari yang sudah dipush diabaikan.

## Do Not / Reject If
- **R1.** Marker user memakai `route_outlet_history.longitude/latitude` (planned) — **reject** karena itu destination, bukan actual user.
- **R2.** Marker user memakai `mobile.visits.arrive_longitude/latitude` join ke `outlet_code` — **reject** karena hanya valid saat arrive outlet, tidak untuk cross-outlet/track.
- **R3.** Initial load empty menghasilkan `current_longitude/latitude` non-zero yang sebenarnya adalah planned — **reject** (fallback salah).
- **R4.** Publisher tidak melakukan validasi range koordinat — **reject** (data kotor akan bocor ke FE).
- **R5.** Consumer tidak ack/nack dengan benar, atau message hilang tanpa DLQ — **reject** (silent data loss).
- **R6.** WebSocket push tidak sampai ke FE karena tidak ada room/per-emp filter — **reject** (broadcast ke room distributor yang berisi emp tsb).
- **R7.** Task menyentuh `live-monitoring-principal` padahal issue ini hanya distributor — **reject** (scope creep).
- **R8.** Task membuat test yang copy credential/JWT — **reject** (SECURITY.md).
