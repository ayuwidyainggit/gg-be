# Plan — SX-2416 & SX-2417: Display Leave Location in Live Monitoring (BE)

> Plan ini adalah source of truth untuk implementasi BE di service `pjp`. Setelah plan
> disetujui, eksekusi source edit dilakukan oleh lane `@orchestrator` + worker
> (`@fixer`/`@backend`) — planner `@artifact-planner` **tidak** mengedit source.

## Goal

Tambahkan dua field response baru, `leave_longitude` dan `leave_latitude`, ke dua endpoint
live monitoring berikut:

- `GET /scylla-pjp/v1/live-monitoring-distributor` (SX-2416)
- `GET /scylla-pjp/v1/live-monitoring-principal` (SX-2417)

Field baru harus muncul di level `destination_data[]`, yaitu level yang sama dengan
`arrive_longitude` dan `arrive_latitude`, agar FE dapat menampilkan titik lokasi saat sales
meninggalkan outlet pada Monitoring Activity / Outlet Details.

Source of truth bukan `mobile.visits`, melainkan langsung dari tabel `outlet_visit_list` yang
memang sudah dipakai endpoint ini untuk field visit lain:

- distributor: `pjp.outlet_visit_list.leave_longitude`, `pjp.outlet_visit_list.leave_latitude`
- principal: `pjp_principles.outlet_visit_list.leave_longitude`, `pjp_principles.outlet_visit_list.leave_latitude`

Dari discovery repo, query existing untuk distributor dan principal **sudah** memiliki `LEFT JOIN`
ke alias `ovl` dari tabel yang tepat. Karena itu shortest-working-diff yang benar bukan refactor,
bukan query baru, dan bukan join ke sumber lain, tetapi hanya menambahkan dua kolom select,
memperpanjang row model + response DTO, lalu memetakan kedua field tersebut ke response JSON.

User juga sudah mengonfirmasi bahwa jika data leave belum ada, response yang diinginkan adalah
`null`, bukan string kosong. Karena kolom DB bertipe varchar/string, bentuk yang paling aman dan
jujur adalah `*string` tanpa `omitempty` pada JSON tag.

## Non-goals

- Tidak menambah endpoint baru.
- Tidak mengubah source `arrive_longitude` / `arrive_latitude` existing.
  - Distributor tetap memakai enrichment `mobile.visits` via `GetDistributorLatestVisitCoordinates`.
  - Principal tetap memakai `ovl.longitude` / `ovl.latitude`.
- Tidak mengubah pagination, ordering, grouping employee/PJP/route, atau shape response selain
  penambahan 2 field baru.
- Tidak menambah dependency baru.
- Tidak melakukan migrasi DB atau alter table.
- Tidak mengubah flow business visit (skip, extra_call, on_hold, finish, attendance, current
  coordinate).
- Tidak menyentuh service lain di monorepo.
- Tidak mengubah docs FE / Google Docs / docs eksternal sebagai bagian dari slice ini.

## Requirements

1. **R1 — distributor endpoint return `leave_longitude`.**
   `GET /api/v1/live-monitoring-distributor` harus mengembalikan `leave_longitude` pada
   `data[].pjp_data[].{route_data, extra_call_data}[].destination_data[]`.

2. **R2 — distributor endpoint return `leave_latitude`.**
   `GET /api/v1/live-monitoring-distributor` harus mengembalikan `leave_latitude` pada level yang
   sama dengan R1.

3. **R3 — principal regular endpoint return `leave_longitude`.**
   `GET /api/v1/live-monitoring-principal` harus mengembalikan `leave_longitude` pada
   `route_data[].destination_data[]`.

4. **R4 — principal regular endpoint return `leave_latitude`.**
   `GET /api/v1/live-monitoring-principal` harus mengembalikan `leave_latitude` pada level yang
   sama dengan R3.

5. **R5 — principal extra-call branch juga ikut.**
   Karena service principal menggabungkan regular rows + extra-call rows, `leave_longitude` dan
   `leave_latitude` juga harus tersedia pada `extra_call_data[].destination_data[]`, bukan hanya
   regular `route_data[]`.

6. **R6 — source of truth distributor.**
   Nilai distributor harus diambil dari `pjp.outlet_visit_list.leave_longitude` dan
   `pjp.outlet_visit_list.leave_latitude` melalui alias `ovl` yang sudah ada di query utama.

7. **R7 — source of truth principal.**
   Nilai principal harus diambil dari `pjp_principles.outlet_visit_list.leave_longitude` dan
   `pjp_principles.outlet_visit_list.leave_latitude` melalui alias `ovl` yang sudah ada di query
   regular dan extra-call.

8. **R8 — no new join source.**
   Worker tidak boleh menambah join ke `mobile.visits` atau sumber lain untuk field leave ini.
   Requirement Jira dan discovery repo sama-sama menunjukkan source yang benar adalah `ovl`.

9. **R9 — nullable behavior.**
   Bila `leave_longitude` / `leave_latitude` belum ada di row `ovl`, response JSON harus emit
   `null`, bukan `""`, sesuai keputusan user.

10. **R10 — empty string normalization.**
    Jika kolom DB berisi string kosong, implementasi harus menormalisasinya menjadi `nil`/`null`
    agar FE tidak menerima nilai kosong ambigu.

11. **R11 — response tag exact name.**
    JSON tag harus persis `leave_longitude` dan `leave_latitude`.

12. **R12 — response level exact placement.**
    Field harus berada di `LiveMonitoringDestinationData`, level yang sama dengan
    `arrive_longitude`, `arrive_latitude`, `leave_at`, `start`, `finish`, dan `skip_at`.

13. **R13 — backward compatible.**
    Tidak ada field existing yang dihapus, di-rename, atau diubah semantik/tipenya.

14. **R14 — distributor arrival path untouched.**
    `GetDistributorLatestVisitCoordinates` dan enrichment `arrive_longitude/arrive_latitude`
    distributor tidak boleh diubah, karena tidak relevan untuk field leave ini.

15. **R15 — principal file_url path untouched.**
    `file_url` subquery principal yang sudah ada tidak boleh direwrite, karena bukan bagian dari
    scope.

16. **R16 — testable slice.**
    Harus ada minimal satu runnable check yang memverifikasi field baru ikut termapping sampai ke
    response structure.

17. **R17 — compile-safe.**
    Build/test package live monitoring pada service `pjp` harus tetap lulus.

18. **R18 — evidence-safe.**
    Verifikasi manual boleh memakai token lokal, tetapi token/cookie/credential tidak boleh masuk
    ke source code, fixture, log evidence, atau plan.

## Acceptance Criteria

- AC1. Query distributor select `leave_longitude` dan `leave_latitude` dari `ovl` dengan alias
  persis `leave_longitude` dan `leave_latitude`.
- AC2. Query principal regular select `leave_longitude` dan `leave_latitude` dari `ovl` dengan
  alias persis yang sama.
- AC3. Query principal extra-call juga select `leave_longitude` dan `leave_latitude` dari `ovl`.
- AC4. `LiveMonitoringDistributorRow` memiliki field `LeaveLongitude *string` dan
  `LeaveLatitude *string`.
- AC5. `LiveMonitoringPrincipalRow` memiliki field `LeaveLongitude *string` dan
  `LeaveLatitude *string`.
- AC6. `LiveMonitoringDestinationData` memiliki field `LeaveLongitude *string` dan
  `LeaveLatitude *string` dengan JSON tag exact, tanpa `omitempty`.
- AC7. `transformDistributorRows` memapping dua field baru ke `destination_data[]`.
- AC8. `transformPrincipalRows` memapping dua field baru ke `destination_data[]` untuk regular dan
  extra-call rows.
- AC9. Saat query memberi empty string atau null, JSON response emit `null`.
- AC10. `arrive_longitude`, `arrive_latitude`, `file_url`, `leave_at`, `start`, `finish`,
  `skip_at`, `skip_reason` existing tidak regress.
- AC11. `rtk go test ./service/live_monitoring/...` lulus dengan assertion baru.
- AC12. `rtk go build ./...` atau minimal package live monitoring pada service `pjp` lulus.

## Constraints

- **Shortest working diff.** Tidak ada abstraction/helper baru jika dua field tambahan di query/model/DTO/mapping sudah cukup.
- **No new join.** Jangan join ke `mobile.visits` untuk leave fields.
- **No behavior rewrite.** Jangan ubah employee-scope pagination, route grouping, extra-call split, attendance enrichment, atau current coordinate logic.
- **JSON exactness.** Field name harus persis `leave_longitude`, `leave_latitude`.
- **Null semantics.** Gunakan `*string` dan `NULLIF(..., '')` agar kosong jadi `null`.
- **Principal dual-path.** Query regular dan query extra-call principal harus sama-sama diupdate.
- **Evidence hygiene.** Token/cookie/env rahasia tidak boleh masuk ke plan, log, atau test fixture.
- **Repo-local docs gap.** `PROJECT_STACK.md` / `PROJECT_COMMANDS.md` / `FRAMEWORK_PLAYBOOK.md` tidak ada; plan harus mengandalkan `.opencode/docs/QUALITY.md`, `AGENTS.md`, dan repo source.

## Risks

- **R-1 (partial principal fix).** Jika worker hanya update `GetPrincipalMonitoring` dan lupa `GetPrincipalExtraCalls`, FE akan melihat field leave hanya pada `route_data` tetapi hilang pada `extra_call_data`.
- **R-2 (string kosong lolos).** Jika query select langsung `ovl.leave_longitude` tanpa `NULLIF`, FE bisa menerima `""` alih-alih `null`.
- **R-3 (wrong type).** Jika field dipaksa `float64`, empty/null di DB akan perlu cast dan berisiko mengubah semantik Jira yang menyatakan varchar.
- **R-4 (scope creep ke arrival).** Karena `arrive_longitude` distributor diisi lewat enrichment dan principal lewat `ovl`, worker bisa tergoda menormalkan semua field lokasi sekaligus. Itu di luar scope.
- **R-5 (manual verification data availability).** Runtime manual hit mungkin tidak menemukan sample outlet dengan leave populated pada tanggal tertentu; karena itu unit-level mapping check wajib ada.
- **R-6 (repo-wide test noise).** Repo punya LSP errors di `mobile/main.go` yang tidak terkait. Validasi harus fokus di module `pjp`, bukan seluruh monorepo root.

## Decisions / Assumptions

- **D1.** Tipe field baru = `*string`.
- **D2.** Query select memakai `NULLIF(ovl.leave_longitude, '') AS leave_longitude` dan `NULLIF(ovl.leave_latitude, '') AS leave_latitude`.
- **D3.** JSON tag tanpa `omitempty` agar `nil` benar-benar emit `null`.
- **D4.** Distributor `GetDistributorLatestVisitCoordinates` tidak disentuh.
- **D5.** Principal regular dan extra-call dua-duanya wajib disentuh.
- **D6.** Google Docs FE contract tidak diverifikasi saat discovery; plan ini mengandalkan prompt user, schema DB, dan source code repo sebagai authority yang cukup.
- **D7.** Tidak perlu `sql.NullString`; `*string` sudah cukup dan lebih konsisten dengan response DTO existing `FileURL`.
