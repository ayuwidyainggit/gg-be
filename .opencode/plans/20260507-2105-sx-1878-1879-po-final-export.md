# Plan — SX-1878/SX-1879 PO Final Qty dan Export Data

## Goal

Implementasikan perbaikan backend di module `sales` untuk:

- `SX-1878`: memastikan process Purchase Order defensif terhadap stock, menyimpan qty processed/final yang benar, dan membuat Purchase Order, Sales Order, serta Final Order konsisten untuk skenario edit qty `> stock`, `< stock`, tanpa edit, dan multi-unit clamp.
- `SX-1879`: memastikan export `GET /sales/v1/download` tidak kosong/null untuk data yang valid dan financial columns di Sales Order tetap konsisten dengan Final Order serta expected QA.

## Non-goals

- Tidak mengubah FE payload contract kecuali perlu menambah validasi error response yang masih kompatibel.
- Tidak hardcode data staging, Jira token, Authorization token, salesman, warehouse, outlet, atau SO number di source/test fixture.
- Tidak mengubah schema/migration kecuali implementer menemukan bukti bahwa kolom sumber qty/amount memang tidak tersedia.
- Tidak rewrite besar arsitektur order/export async report.
- Tidak memakai token dari Jira evidence di source, tests, logs, atau artifacts.

## Requirements

### SX-1878

- BE tidak boleh mempercayai raw FE payload jika qty request melebihi available stock aktual.
- Semua qty process harus dinormalisasi ke base/smallest unit sebelum dibandingkan dengan stock.
- Available stock untuk edit order yang sudah punya alokasi harus dihitung sebagai current warehouse stock plus existing allocated qty untuk detail/order yang sama, agar reprocess tidak salah membatasi alokasi yang sudah pernah dipakai.
- Qty processed final harus disimpan ke kolom yang dibaca oleh Purchase Order, Sales Order, dan Final Order.
- Final Order harus memakai qty processed/final hasil enhance, bukan original mobile order atau stale fallback.
- `qty_change* == 0` tidak boleh dianggap sebagai bukti “tidak ada edit”; `purchase_order.qty_po*` pada payload process harus menjadi source of truth.
- WH stock dan On Cust Order harus konsisten antar tab/detail setelah process.

### SX-1879

- Export Purchase Order tidak boleh kosong jika data filter valid memang ada.
- Sales Order export untuk SO sekelas `SO2605060003` harus menghasilkan financial values sesuai expected QA.
- Final Order export tetap pass.
- Decimal/thousands separator Indonesia tetap applied.
- Sales Order dan Final Order memakai formula/mapping yang konsisten kecuali ada business rule eksplisit berbeda.

## Acceptance Criteria

### SX-1878

- Edit Qty `> Stock` tetap pass.
- Edit Qty `< Stock` pass di Purchase Order, Sales Order, dan Final Order.
- Without Edit Qty tetap pass.
- Case FE available stock `4 0 3` vs edited order `4 0 4`: BE tidak memproses lebih dari available stock.
- WH Stock dan On Cust Order benar dan konsisten di semua tab/detail.
- Unit conversion tidak menyebabkan over-allocation.

### SX-1879

- Export Purchase Order tidak null untuk filter yang memiliki data.
- Sales Order export untuk fixture sekelas `SO2605060003` memiliki `Discount`, `Net Sales (ExcPPN)`, `PPN`, dan `Gross` sesuai expected QA.
- Final Order export tetap pass.
- Decimal separator ribuan Indonesia tetap applied.
- Regression tests export berhasil.

## Risks

- Definisi “available stock” bisa ambigu untuk order yang sudah processed. Rekomendasi plan: clamp terhadap `current stock + existing allocation for same order detail` agar edit existing allocation tidak dihitung sebagai over-allocation palsu.
- `FindWarehouseStockByWhIdAndProIds` dan `GetCurrentStock` bisa berbeda sumber/semantik; implementer harus konsisten memilih sumber untuk clamp dan display.
- Mengubah filter PO export yang sebelumnya mengecualikan blank `po_no` dapat menambah rows; perlu regression test agar hanya rows order valid yang masuk.
- Financial formula export bisa berbeda antara domain “gross before tax” dan “total after tax”; QA expected menunjukkan `Gross` sama dengan net/gross sales sebelum PPN pada case evidence.
- Staging reproduction tergantung data yang mungkin berubah setelah retest QA.

## Decisions/Assumptions

- Interaction level: Assumption-first. Requirements cukup detail; plan dapat dilanjutkan dengan asumsi eksplisit.
- Question Gate: tidak ada pertanyaan yang harus menghentikan penulisan plan, tetapi implementer harus memvalidasi dua asumsi domain sebelum merge:
  1. Untuk edit existing processed order, available stock = current warehouse stock + existing `qty_final`/allocated qty detail yang sama.
  2. PO export harus memakai fallback `order_no` saat `po_no` kosong jika row tersebut memang berasal dari order PO/mobile yang valid; jangan drop row hanya karena `po_no` blank.
- Qty source of truth untuk process PO adalah normalized/clamped `purchase_order.qty_po*`.
- Saat PO tab diedit, hasil clamped harus cascade ke `qty_po*`, `qty*`, dan `qty*_final` kecuali user secara eksplisit mengedit Final Order tab pada request yang sama.
- Untuk SX-1879, existing code sudah mengandung sebagian fix; implementasi berikut harus lebih banyak berupa verifikasi gap dan regression hardening.

### Green Step

- Add a helper in `order_service.go` or small service helper file to:
  1. Convert requested qty1/qty2/qty3 to base qty.
  2. Resolve available base qty from current stock plus existing allocation where applicable.
  3. Clamp base qty to available.
  4. Convert clamped base qty back to normalized qty1/qty2/qty3.
- Apply helper to `UpdateEnhance` PO path and add-product PO path before building update maps/stock updates.
- Ensure final fields are written from clamped qty in same transaction.
- Adjust DetailV2 fallback only if test proves final display can still fall back to stale/original values.
- For SX-1879, keep existing mapper/formatter behavior if tests already pass; only change PO filter/fallback or formula gaps identified by Red tests.

### Refactor Step

- Extract repeated clamp/normalization code into well-named functions after tests pass.
- Avoid duplicating conversion arithmetic outside `conversion.QtyUnit`/`conversion.Qty`.
- Keep export formula helper small if SO/Final financial calculations are duplicated.

### Edge Cases

- Nil `MpConvUnit2`/`MpConvUnit3`; default safely to `1` only if existing project helper does so.
- Zero stock; processed qty should become zero unless business rule allows backorder.
- Existing allocated qty greater than current stock due to previous process; edit lower should still be allowed up to existing allocation if using current + existing allocation.
- Product with only smallest unit (`convUnit2=1`, `convUnit3=1`).
- Negative qty should be rejected or normalized according to existing validation; do not silently allocate negative stock.
- Multiple details for same product/order; do not double-count existing allocation from unrelated rows unless stock system treats all same-order rows together.
- Promo product rows (`item_type != 1`) should not be accidentally clamped as normal item unless existing behavior requires it.

## Implementation Steps

1. Reconfirm route wiring and request structs for `EditOrderEnhanceBody`, `EditPurchaseOrderDetail`, `AddPurchaseOrderDetail`, and final/detail response structs.
2. Add Red tests for SX-1878 in `sales/service/order_service_test.go`.
3. Implement qty clamp helper:
   - Inputs: requested qty1/qty2/qty3, conversion units, current stock base qty, existing allocated base qty.
   - Output: clamped base qty and normalized qty unit breakdown.
   - Use `conversion.QtyUnit.ToTotalQuantity()` and `conversion.Qty.ConvToQtyConversion()`.
4. In `UpdateEnhance` Purchase Order path:
   - Fetch current stock with `StockRepository.GetCurrentStock` when `ro.WhId` exists.
   - Resolve existing allocated qty from `existingDetail.QtyFinal` or relevant processed qty.
   - Clamp requested base qty.
   - Persist clamped qty to PO/SO/Final fields.
   - Use clamped base qty for `SalesOrderStockUpdate.QtyOrder`.
   - Recompute amounts/VAT/discount using clamped qty, not raw request qty.
5. Apply same clamp behavior to `createOrderDetailFromPurchaseOrder` for add PO rows. Existing allocation is zero for new rows.
6. Review `ProcessEnhanceWithoutProductEdit`:
   - If no edit scenario only changes status, leave detail rows untouched.
   - If final fields may be null/stale, add safe sync from existing processed qty only when it does not overwrite explicit final edits.
7. Review `DetailV2` Final Order construction:
   - Ensure `DetailsFinal.Normal` uses `qty*_final` for display and stock add-back.
   - Remove or adjust fallback to original `qty*` only when final fields are nil and order has never been processed.
8. Add/adjust SX-1879 tests in `sales/service/so_service_test.go`:
   - Verify SO/Final financial consistency for same fixture.
   - Verify PO export fallback does not drop valid rows with blank `po_no` if `order_no` exists.
9. If Red test proves current PO filter is too strict, update `filterDownloadDataPoWithPONumber` or repository query mapping so valid PO rows use `order_no` fallback and are not filtered out.
10. Run targeted tests, then full module tests.
11. Manual verification if credentials/env are available:
    - Start `sales` service or use staging base URL.
    - Reproduce enhance for safe test order only.
    - Verify `GET /sales/v2/orders/{so_no}` for Purchase/Sales/Final sections.
    - Trigger export query for QA date/salesman and inspect report output.
