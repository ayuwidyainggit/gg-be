# SX-2090 Promo Flag Persist Plan

## Goal
Perbaiki `PATCH /sales/v1/orders/enhance/:order_no` agar `is_product_promotion_so` dari FE tersimpan persis ke `sls.order_detail.is_product_promotion_so` untuk update existing detail dan insert `add_sales_order`, termasuk eksplisit `false`.

## Non-goals
- Tidak ubah kontrak endpoint selain perilaku persist flag promo sales.
- Tidak ubah schema DB kecuali validasi lokal membuktikan kolom hilang.
- Tidak pakai token, credential, payload produksi, atau data Jira mentah.
- Tidak ubah promo engine rules di luar kebutuhan preserve explicit FE flag.

## Scope
Target module `sales`.

Flow yang wajib dicek dan diperbaiki:
- `sales/controller/order_controller.go` handler `UpdateEnhance`.
- `sales/entity/edit_order_enhance.go` DTO enhance.
- `sales/service/order_service.go` mapping update, mapping insert, promo recompute override.
- `sales/repository/order_repository.go` partial update persistence.
- `sales/service/order_service_test.go` regression tests.

## Requirements
- `sales_order[]` dengan `order_detail_id` dan `is_product_promotion_so:false` harus menghasilkan update DB `is_product_promotion_so = false`.
- `sales_order[]` dengan `is_product_promotion_so:true` harus menghasilkan update DB `is_product_promotion_so = true`.
- `add_sales_order[]` dengan `is_product_promotion_so:false` harus insert row dengan `is_product_promotion_so = false` dan tidak ditimpa recompute promo setelah insert.
- `add_sales_order[]` dengan `is_product_promotion_so:true` harus insert row dengan `is_product_promotion_so = true` dan tidak ditimpa recompute promo setelah insert.
- Explicit `false` harus diperlakukan sebagai value valid karena DTO memakai `*bool`.
- Omitted field tetap mengikuti perilaku existing/default; jangan paksa breaking change tanpa kebutuhan.

## Acceptance Criteria
- `PATCH /sales/v1/orders/enhance/:order_no` menerima `is_product_promotion_so:false`.
- Existing order detail row update ke `false` saat payload false.
- Existing order detail row update ke `true` saat payload true.
- New order detail row insert ke `false` saat payload false.
- New order detail row insert ke `true` saat payload true.
- Tidak ada fallback/default/recompute yang menimpa explicit `false`.
- Unit/service tests menutup `sales_order` dan `add_sales_order` paths.
- Tidak ada secret/token/credential baru dalam tests, logs, docs, fixtures.

## Existing Patterns/Reuse
- Reuse DTO pointer bool:
  - `EditSalesOrderDetail.IsProductPromotionSo *bool json:"is_product_promotion_so,omitempty"`
  - `AddSalesOrderDetail.IsProductPromotionSo *bool json:"is_product_promotion_so,omitempty"`
- Reuse `normalizeEnhancePromoFlags` untuk alias `is_product_promotion` → `is_product_promotion_so`.
- Reuse `explicitPromoOverrides` + `applyExplicitPromoFlagOverride` agar recompute tidak menimpa FE explicit flag.
- Reuse `UpdateDetailPartial` GORM `Updates(map[string]interface{})`; map update aman untuk boolean `false`.
- Reuse test mocks di `sales/service/order_service_test.go`.

Tidak perlu util KiloCode baru; pola project lokal sudah cukup.

## Constraints
- Repo rule: mulai runtime check dari `rtk docker compose -f docker-compose.yml ps`; sudah dicoba, tool menangkap warning compose `version` obsolete. Rerun bila perlu DB/runtime evidence.
- Semua Go command di module `sales`, bukan repo root.
- Shell workflow repo ini pakai prefix `rtk`.
- Layering tetap Controller → Service → Repository → DB.
- Write tetap dalam service transaction; repository tetap tx-aware via context.
- Semua query/write tetap pakai `cust_id` filter.

## Risks
- Jira menyebut DB field expected `is_product_promotion`, tapi schema/code punya `is_product_promotion_so`; implementasi harus pakai field aktual sales tab.
- Promo recompute dapat menulis ulang `is_product_promotion_so` setelah insert jika explicit override untuk row baru tidak dicatat.
- Integration test full endpoint bisa berat karena seed order/promo/stock kompleks; service-level regression wajib, DB integration optional bila harness tersedia.
- `findOrderDetailsForProforma` snapshot perlu mencerminkan inserted row agar override ID cocok.

## Decisions/Assumptions
- Judul Jira + expected condition dipakai sebagai source of truth: FE `false` → DB `false`.
- Kolom target untuk endpoint sales tab adalah `sls.order_detail.is_product_promotion_so`.
- DTO pointer bool dipertahankan; tidak diubah ke non-pointer karena omission vs explicit false masih berguna.
- Bug paling mungkin: insert path `add_sales_order[]` sudah menyimpan pointer flag saat `StoreDetail`, tetapi belum menambahkan `explicitPromoOverrides` untuk row baru, sehingga recompute bisa overwrite flag.
- Tidak ada pertanyaan material tersisa. Plan bisa langsung dieksekusi.

Red step:

   - `json.Unmarshal([]byte({"sales_order":[{"order_detail_id":6960,"is_product_promotion_so":false}],"add_sales_order":[{"pro_id":8438,"is_product_promotion_so":false}]}), &entity.EditOrderEnhanceBody{})`
   - Assert pointers non-nil dan values false.
2. Tambah test existing update explicit false dan true:
   - Call `UpdateEnhance` dengan dua `sales_order` rows, one false one true.
   - Capture `UpdateDetailPartial` updates for each `order_detail_id`.
   - Assert update map contains `is_product_promotion_so` with exact bool.
3. Tambah test `createOrderDetailFromSalesOrder` explicit false:
   - Existing test covers true; add false case or table-driven true/false.
   - Assert stored `OrderDetail.IsProductPromotionSo` pointer non-nil and exact bool.
4. Tambah failing regression for `add_sales_order` recompute preserve false/true:
   - `storeDetailFn` assigns inserted IDs.
   - `findOrderDetailsForProformaFn` returns inserted rows.
   - Promo mock returns promo outcome likely `true` for inserted row.
   - Request sends `add_sales_order` explicit false for one row and true for another where feasible.
   - Assert final snapshot `UpdateDetailPartial` for inserted IDs keeps exact explicit flags.

Green step:
- In `UpdateEnhance` add sales order insert loop, after `createOrderDetailFromSalesOrder` and successful `stockUpdate`, record:
  - if `addDetail.IsProductPromotionSo != nil`, set `explicitPromoOverrides[stockUpdate.RefDetId].SalesOrder = boolPtr(*addDetail.IsProductPromotionSo)`.
- Keep existing `StoreDetail` assignment `IsProductPromotionSo: addDetail.IsProductPromotionSo`.
- Do not add truthy guard.

Refactor step:
- If tests duplicate setup too much, extract tiny test helper for bool pointer assertions/capture only inside test file.
- Avoid broad service rewrite.

Edge cases:
- Explicit false with qty/price update in `sales_order[]`.
- Explicit true with qty/price update in `sales_order[]`.
- Explicit false in `add_sales_order[]` while promo consult marks product as promo.
- Explicit true in `add_sales_order[]` while promo consult/default would be false.
- Omitted `is_product_promotion_so` remains current/default behavior.
- Generic alias `is_product_promotion` still maps via `normalizeEnhancePromoFlags`.
