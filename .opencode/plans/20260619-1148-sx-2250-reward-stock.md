## Goal

Pastikan API Detail Sales Order `GET /sales/v2/orders/{order_no}` mengembalikan `qty1_stok`, `qty2_stok`, `qty3_stok` benar untuk row normal dan row reward promotion di:

- `details.normal`
- `details_final.normal`

Formula wajib:

```text
stock_display_per_row = warehouse_stock_current + qty_order_current_row
```

Row dengan `pro_id` sama tidak boleh saling mengkontaminasi stock display.

- Tidak copy credential, bearer token, curl auth, atau data sensitif Jira ke source, test, log, commit, atau artifact.
- Tidak ubah promo eligibility, promo reward calculation, atau `aggregatePromoByProduct*` kecuali test membuktikan regression langsung.
- Tidak ubah stock movement/on-customer stock jika current branch sudah benar.
- Tidak ubah schema/model/entity field names.
- Tidak ubah service lain di luar `sales/`.

## Scope

In scope:

- `DetailV2` stock display untuk `details.normal` dan `details_final.normal`.
- Unit test same `pro_id` normal row + reward row.
- Unit test same `pro_id` qty berbeda antar-row.
- Validasi existing promo flag behavior dari SX-2090/SX-2129 tetap aman.

## Requirements

1. `details.normal[].qty1_stok`, `qty2_stok`, `qty3_stok` dihitung dari warehouse current product stock + qty row itu saja.
2. `details_final.normal[].qty1_stok`, `qty2_stok`, `qty3_stok` memakai rule sama, dengan qty final row terkait.
3. Row normal (`item_type=1`) dan reward (`item_type=2`) dengan `pro_id` sama tetap muncul sebagai row terpisah.
4. Reward row tidak boleh di-skip dari stock display hanya karena `is_product_promotion_so=true` atau `is_product_promotion_final=true`.
5. Normal row tidak boleh menerima promo/reward qty row lain.
6. Flags tetap benar:
   - normal ordered product: `is_product_promotion_so=false`, `is_product_promotion_final=false`
   - reward product: `is_product_promotion_so=true`, `is_product_promotion_final=true`
7. Jika stock movement process SO masih mengecualikan reward product, fix minimal supaya reward qty masuk on-customer stock. Jika tidak terbukti, jangan ubah.
