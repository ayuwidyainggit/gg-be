# Plan SX-2154 — Enhance `POST /sales/v1/orders` order_type

Readiness: `ready-for-implementation`
Plan Quality Gate: `PASS`
Mode: Maintenance Stability Mode

## Goal

Tambahkan dukungan `order_type` pada `POST /sales/v1/orders`, simpan ke `sls.order.order_type`, dan untuk `order_type = "O"` simpan original taking order qty ke `sls.order_detail.original_qty_po1/2/3` tanpa mengubah flow existing untuk `SO` atau payload lama.

## Non-goals

- Tidak ubah GET detail, promo injection, proforma, approval, invoice, atau endpoint lain.
- Tidak ubah kalkulasi qty, stock, subtotal, discount, promo, tax, status, response shape.
- Tidak implement flow Canvas baru selain accept dan persist `C`.
- Tidak ubah `pjp-sales` atau `mobile` kecuali compile gagal karena shared-copy expectation eksplisit.

## Scope

- Service target: `sales`.
- Endpoint target: `POST /sales/v1/orders`.
- DB migration additive untuk `sls.order` dan `sls.order_detail`.
- DTO/model/service mapper/repository persistence via existing GORM create.
- Unit tests mapper/service dan minimal repository/integration bila fixture DB tersedia.

## Requirements

- `CreateOrderBody` punya optional `OrderType *string` atau nullable-equivalent dengan JSON `order_type`.
- Valid values: `O`, `C`, `SO`; optional supaya backward compatible.
- `model.Order` punya `OrderType *string` dengan `gorm:"column:order_type" json:"order_type"`.
- `model.OrderDetail` punya `OriginalQtyPo1/2/3 *float64` dengan column tags.
- `CreateOrderDetBody` perlu menerima `qty_po1/2/3`, karena AC payload memakai field itu.
- Pada `order_type == "O"`, detail normal dan promo yang disimpan harus set `OriginalQtyPo1 = QtyPo1`, `OriginalQtyPo2 = QtyPo2`, `OriginalQtyPo3 = QtyPo3` dari request detail.
- Pada `order_type == "SO"`, `"C"`, atau nil, jangan set original qty kecuali payload/model existing melakukan itu; target aman: biarkan nil.

## Acceptance Criteria

- AC1: Request dengan `order_type = "O"` lolos parse/validasi.
- AC2: Row `sls.order` menyimpan `order_type = 'O'`.
- AC3: Row `sls.order_detail` menyimpan `original_qty_po1/2/3` dari `qty_po1/2/3` untuk `order_type = "O"`.
- AC4: `order_type = "SO"` dan tanpa `order_type` mempertahankan behavior existing.
- AC5: Payload lama tanpa `order_type` tetap sukses.

## Existing Patterns/Reuse

- Reuse `OrderController.Create` parse/validate flow di `sales/controller/order_controller.go`.
- Reuse `structs.Automapper` dari request ke `model.Order` dan detail ke `model.OrderDetail`.
- Reuse tx-aware repository `Store` dan `StoreDetail`; tidak perlu raw insert baru.
- Reuse validation tag `validate:"omitempty,oneof=O C SO"` sesuai pattern enum optional di `sales/entity/*`.
- Reuse migration style `BEGIN; ALTER TABLE IF EXISTS ... ADD COLUMN IF NOT EXISTS ... COMMENT ON COLUMN ... COMMIT;`.

## Constraints

- Repo rule: Controller → Service → Repository → DB.
- Write operations tetap di service transaction.
- Target validasi dari folder `sales`.
- Migration command untuk `sales` tidak documented; SQL harus idempotent.
- `sls.order` style migration existing tidak quote table; ikuti existing `sls.order`.

## Risks

- `CreateOrderDetBody` belum punya `qty_po1/2/3`; kalau tidak ditambah, AC payload tidak masuk model.
- Integration test create order dapat berat karena perlu outlet/product/discount/stock fixtures.
- `SO` regression risk tinggi karena `Store` banyak logic promo/discount/stock; perubahan harus sempit pada field tambahan.
- Jika DB sudah punya constraint berbeda untuk enum, migration varchar+check bisa konflik; cek actual DB sebelum apply bila runtime tersedia.

## Decisions/Assumptions

- Tipe DB `order_type`: `VARCHAR(2) NULL` dengan `CHECK (order_type IS NULL OR order_type IN ('O','C','SO'))`, karena migration existing memakai varchar/char dan tidak ditemukan PostgreSQL enum pattern di `sales`.
- `original_qty_po1/2/3`: `FLOAT4 NULL` sesuai Jira/docs.
- Validasi invalid enum: reject dengan 400 karena project punya `oneof` validation pattern dan value list eksplisit.
- Source original qty: `CreateOrderDetBody.QtyPo1/2/3`, bukan `Qty1/2/3`, karena AC mapping eksplisit `qty_po1/2/3 -> original_qty_po1/2/3`.
- Unanswered material question: tidak ada. Jika user ingin invalid enum tidak direject, ubah validation jadi no-op dan biarkan DB check menolak; namun plan memilih safer project pattern.

1. Add migration file under `sales/migration/sls.order/` or `sales/migration/sls.order_detail/`, preferably `sales/migration/sls.order/add_order_type_and_original_qty_po_fields.sql`, containing both table changes for single issue.
2. Add rollback file matching repo pattern if desired: drop check constraint then columns.
3. Update `sales/entity/order.go`: add `OrderType *string 'json:"order_type" validate:"omitempty,oneof=O C SO"'` to `CreateOrderBody`; optionally add to response/list structs only if select response must expose it.
4. Update `sales/entity/order_detail.go`: add `QtyPo1/2/3 *float64 'json:"qty_po1"'` to `CreateOrderDetBody`.
5. Update `sales/model/order.go`: add `OrderType *string 'gorm:"column:order_type" json:"order_type"'` to `Order` and `OrderList` if reads need it.
6. Update `sales/model/order_detail.go`: add `OriginalQtyPo1/2/3 *float64` to `OrderDetail`; add to read struct only if endpoint returns it.
7. Update `sales/service/order_service.go`: after `structs.Automapper(detail, &gdsDetModel)`, call helper when `request.OrderType != nil && *request.OrderType == "O"` for normal and promo detail paths.
8. Add unit tests before production changes if possible; then update expected fixtures.
9. Run targeted tests then all `sales` tests.
10. If local DB available, apply migration on local DB and smoke create order with `order_type = "O"`; assert DB rows.
