# Plan SX-2079 Dropdown Scope Region, Area, Business Unit

Task id: `20260526-1537-sx-2079-dropdown-scope`

## Goal

Ubah API dropdown `master` agar user principal melihat data sesuai scope employee dari `mst.m_employee`: `region_scope`, `area_scope`, `distributor_scope`, plus mapping table employee. Perubahan harus defensif terhadap `Specific`, `SPESIFIC`, `SELECTED`, `All`, `ALL`, dan `NULL`.

## Non-goals

- Tidak mengubah behavior user distributor/non-principal di luar pengamanan scope principal.
- Tidak membuat migration baru kecuali implementasi menemukan kolom/tabel belum ada di DB target.
- Tidak mengubah response contract selain isi data dropdown.
- Tidak memperbaiki fitur lain seperti employee PJP atau monitoring.

## Scope

Target module: `master`.

Endpoint target:

- `GET /master/v1/regions`
- `GET /master/v1/areas`
- `GET /master/v1/business-unit`

Layer target:

- Controller: tambah data auth context ke filter.
- Service: deteksi principal dan load scope employee sekali per request.
- Repository: query scoped region, area, business-unit/distributor.
- Tests: unit/query-builder/service/controller sesuai matrix.

## Requirements

- Principal ditentukan dari `distributor_id == nil || distributor_id == 0` sesuai pattern `BusinessUnitService.GetBusinessUnit`.
- Current employee id diambil dari Fiber local `employee_id`, bukan payload.
- Scope employee diambil dari `mst.m_employee` berdasarkan `emp_id` dan `cust_id` current principal.
- `normalizeScope` harus mengembalikan `specific` untuk `specific`, `spesific`, `selected` dalam case apa pun; nilai lain, kosong, dan null menjadi `all`.
- Mapping join memakai kolom `emp_id` berdasarkan migration dan existing repository.
- Mapping table harus filter `is_del = false`.
- Master table harus filter `is_del = false` dan `is_active` sesuai payload.
- Query join mapping harus pakai `DISTINCT` atau `COUNT(DISTINCT ...)` agar tidak duplicate.
- Existing query params `q`, `page`, `limit`, `sort`, `is_active`, `region_id`, `area_id` tetap jalan.
- Jika `region_id` / `area_id` dikirim, filter payload tetap dipakai tetapi principal scope tetap enforced agar tidak bypass.

## Acceptance Criteria

- `GET /master/v1/regions` untuk principal `region_scope=specific/spesific/selected` hanya mengembalikan mapped regions.
- `GET /master/v1/regions` untuk principal `region_scope=all/null` mengembalikan region berdasarkan employee `cust_id`.
- `GET /master/v1/areas` tanpa `region_id` untuk principal `area_scope=specific/spesific/selected` hanya mengembalikan mapped areas.
- `GET /master/v1/areas` tanpa `region_id` untuk `area_scope=all/null` dan `region_scope=all/null` mengembalikan area berdasarkan employee `cust_id`.
- `GET /master/v1/areas` tanpa `region_id` untuk `area_scope=all/null` dan `region_scope=specific/spesific/selected` mengembalikan area dalam mapped regions.
- `GET /master/v1/business-unit` tanpa `region_id` dan `area_id` untuk `distributor_scope=specific/spesific/selected` hanya mengembalikan mapped distributors.
- `GET /master/v1/business-unit` tanpa `region_id` dan `area_id` untuk semua scope `all/null` mengembalikan distributor berdasarkan current employee `parent_cust_id`.
- `GET /master/v1/business-unit` mengikuti fallback region/area mapping saat distributor scope `all/null` dan region/area scope specific.
- `q`, pagination, sort, `is_active`, `region_id`, `area_id` tetap berlaku.
- Non-principal behavior tetap sama.
- Dropdown tidak berisi duplicate rows.

Tidak ada KiloCode/project utility siap pakai untuk dropdown scope enforcement; perlu extend repository/service existing.

## Constraints

- Shell validation di repo ini harus pakai prefix `rtk`.
- Validasi dijalankan dari `master/`, bukan repo root.
- Tenant filter wajib: `cust_id` untuk master principal, `parent_cust_id` untuk distributor/business-unit child data.
- Jangan copy token dari docs ke test/log.
- Existing repository region/area memakai string SQL concatenation; perubahan baru sebaiknya memakai args agar lebih aman, minimal untuk query baru.
- Sort field raw dari request sudah existing; jangan perluas risiko. Jika disentuh, tambahkan allowlist kecil untuk endpoint target.

## Risks

- Existing `m_area_repository.go` search branch memakai alias `a` padahal query alias `ma`; test `q` perlu menangkap ini saat area query disentuh.
- Existing business-unit query memakai `EXISTS smc.m_customer`; requirement memakai `mst.m_distributor.parent_cust_id`. Implementasi aman: filter utama `md.parent_cust_id = ?`, dengan kompatibilitas legacy `OR EXISTS (...)` hanya jika local data/test menunjukkan `md.parent_cust_id` belum lengkap.
- Jika `employee_id` JWT principal kosong/0, scope tidak bisa diterapkan. Return bad request atau record not found sesuai pattern error service; jangan fallback all diam-diam.
- Existing `RegionQueryFilter` dan `AreaQueryFilter` mungkin belum robust untuk `region_id[]`/`area_id[]`. Kalau FE mengirim bracket arrays, controller perlu normalisasi seperti business-unit.
- Menambah dependency repository ke service bisa memecah stub tests; update stubs secara eksplisit.

## Decisions/Assumptions

- `Specific`, `SPESIFIC`, dan `SELECTED` diperlakukan sama: `specific`.
- `All`, `ALL`, empty, unknown, dan null diperlakukan sebagai `all`.
- Principal-only rule aktif bila `distributor_id` dari token nil/0. Untuk region/area controller, tambahkan `DistributorId` ke filter dari local token.
- Scope tetap enforced meskipun request mengirim `region_id` atau `area_id`; payload filter menjadi intersection, bukan bypass.
- Current employee lookup memakai `emp_id` dari JWT local `employee_id` dan `cust_id` principal.
- Business-unit master adalah `mst.m_distributor`, sesuai existing code and docs.
- Tidak ada pertanyaan blocker; ambiguity query explicit filter diselesaikan dengan default aman: intersection dengan principal scope.

## Implementation Steps

1. Add shared dropdown scope model/helper in `master/entity` or `master/service`.
   - Prefer `master/entity/dropdown_scope.go` for data structs and `master/service/dropdown_scope.go` for normalization/service logic if no package cycle.
   - Helper:

```go
func NormalizeDropdownScope(value string) string {
	v := strings.ToLower(strings.TrimSpace(value))
	switch v {
	case "specific", "spesific", "selected":
		return "specific"
	default:
		return "all"
	}
}
```

2. Add employee scope repository method.
   - Recommended interface method: `FindEmployeeDropdownScope(empID int, custID string) (model.EmployeeDropdownScope, error)`.
   - Select only `emp_id`, `cust_id`, `region_scope`, `area_scope`, `distributor_scope` from `mst.m_employee` where `emp_id = $1`, `cust_id = $2`, `is_del = false`.
   - Parent customer comes from JWT `parent_cust_id`; employee row supplies `cust_id` and scopes.

3. Extend query filter structs.
   - `RegionQueryFilter`: add `EmployeeId int`, `DistributorId int`, optional `EmployeeScope entity.EmployeeDropdownScope` only if repository needs normalized values.
   - `AreaQueryFilter`: same.
   - `BusinessUnitQueryFilter`: add `EmployeeId int`, maybe normalized scope fields.

4. Update controllers.
   - Region/Area: set `EmployeeId` from `c.Locals("employee_id")`, `DistributorId` from `c.Locals("distributor_id")` if present.
   - Business-unit already has principal convention; also set `EmployeeId` and `DistributorId` from token when query param missing.
   - Normalize `region_id[]` and `area_id[]` for region/area if tests show `QueryParser` does not parse bracket arrays.

5. Update services.
   - Inject employee scope repository or a small `EmployeeScopeRepository` into region, area, business-unit services.
   - For principal: load employee scope once, normalize three scopes, attach to filter or call scoped repository method.
   - For non-principal: keep existing calls unchanged.

6. Update region repository.
   - Add scoped query path or change `FindAllByCustId`/`LookupMode` to call query builder with scope.
   - Specific: join `mst.m_employee_region_mapping erm ON erm.region_id = a.region_id AND erm.cust_id = ? AND erm.emp_id = ? AND erm.is_del = false`.
   - All: filter `a.cust_id = employee.CustId`.
   - Preserve `region_id`, `q`, `is_active`, sort, paging.
   - Use `COUNT(DISTINCT a.region_id)` and `SELECT DISTINCT ...` when mapping join active.

7. Update area repository.
   - Specific area: join `mst.m_employee_area_mapping eam` by `area_id`, `cust_id`, `emp_id`, `is_del=false`.
   - All area + all region: filter `ma.cust_id = employee.CustId`.
   - All area + specific region: join `mst.m_employee_region_mapping erm` by `ma.region_id`.
   - Explicit `region_id` remains as extra `ma.region_id IN (?)` filter.
   - Fix alias in search branch from `a` to `ma` if touched.
   - Use distinct count/select when join active.

8. Update business-unit repository.
   - Specific distributor: join `mst.m_employee_distributor_mapping edm` by `distributor_id`, `cust_id`, `emp_id`, `is_del=false`.
   - All/all/all: filter by current employee/token `parent_cust_id`. Prefer `md.parent_cust_id = ?`; keep `EXISTS smc.m_customer` compatibility only if needed by existing data/tests.
   - Region+area specific fallback: join both `mst.m_employee_region_mapping erm` and `mst.m_employee_area_mapping eam`.
   - Area specific fallback: join `eam` only.
   - Region specific fallback: join `erm` only.
   - Preserve explicit `region_id` and `area_id` filters as intersection.
   - Use `COUNT(DISTINCT md.distributor_id)` and `SELECT DISTINCT ON (md.distributor_id)` or `SELECT DISTINCT`.

9. Update tests first, then code to pass.

10. Run validation commands and API smoke with local tokens/users if available.
