# SX-2090 / SX-2129 Promo Flag dan Same SKU Reward Plan

## Goal
Perbaiki `sales` backend agar promo flag eksplisit dari FE, detail SO/final, dan reward product tetap konsisten untuk kasus same SKU normal product + reward product.

Target utama:
- `is_product_promotion_so=false` dari payload tetap tersimpan sebagai `false`, bukan dianggap kosong.
- Normal row `item_type=1` dan reward row `item_type=2` dengan `pro_id` sama tetap terpisah.
- Final fields mengikuti SO fields saat belum ada proforma.
- MR !95 eligibility filter tidak regresi.

## Non-goals
- Tidak ubah schema DB kecuali implementer menemukan bukti kolom hilang.
- Tidak ubah promo engine V2 rule di luar row-aware distribution dan reward row separation.
- Tidak copy bearer token, password, curl auth, atau data credential dari Jira ke repo.
- Tidak merge row reward ke normal row walau `pro_id` sama.
- Tidak set `is_product_promotion_po=true` untuk SO reward row tanpa rule domain eksplisit.

## Requirements
- Create order flow (`Store`) must keep normal ordered row and reward row separate when ordered SKU equals reward SKU.
- Field exists + value `true` persists `true`.
- Field missing uses existing fallback/recompute logic.
- Explicit `false` survives existing detail update, add detail insert, recompute, final sync, and detail readback.
- For same SKU normal + reward:
  - `item_type=1` remains normal product row.
  - `item_type=2` remains reward product row.
  - Reward product promo flag/remarks/amount must not contaminate normal row only because `pro_id` matches.
- For rows/orders without proforma, final promo fields follow SO promo fields:
  - `is_product_promotion_final = is_product_promotion_so`
  - `promo_final1..5 = promo_so1..5`
  - `promo_remarks_final = promo_remarks_so`
- Purchase reward flag stays false unless PO-specific flow sets it explicitly.
- Eligibility filter from MR !95 remains: reward product promo from ineligible promo does not attach to normal detail snapshot.

## Acceptance Criteria
- BE persists explicit `is_product_promotion=false` / `is_product_promotion_so=false` from FE.
- Explicit false survives insert/update/recompute/final sync.
- Explicit true stays true.
- Missing field still uses current fallback/recompute behavior.
- `pro_id=8435` latest case can be represented as two separate rows:
  - `item_type=1`: `is_product_promotion=false`, `is_product_promotion_so=false`, `is_product_promotion_po=false`, `is_product_promotion_final=false`
  - `item_type=2`: `is_product_promotion=true`, `is_product_promotion_so=true`, `is_product_promotion_po=false`, `is_product_promotion_final=true`
- Final promo fields sync from SO only when safe: no proforma.
- Existing SX-2090 eligibility regression stays valid for `pro_id=8438`: no `testbarang` amount/remarks from ineligible reward promo.
- Tests cover explicit false, explicit true, missing flag, same SKU normal+reward, final sync, and ineligible promo aggregation.

## Risks
- `distributePromoToDetailRowsV2` currently uses product-level aggregate and can OR reward product `IsProductPromotion` into normal detail rows with same SKU.
- `SyncFinalOrderFields` currently does not sync promo component fields or promo flags, so final can diverge from SO.
- `OrderDetailRead` lacks `is_proforma_inv`; final sync safety may require repository SQL condition using order/header table or adding read field if existing query supports it.
- `hasPersistedPromoSnapshot` uses non-pointer response bool, so explicit false alone does not count as snapshot. Do not use that helper to prove false persistence.
- Changing aggregate key globally may break response injection if not migrated carefully.

## Decisions/Assumptions
- Source of truth is latest user-provided Jira/PM expected behavior.
- Local code evidence is enough; no public docs needed.
- Same SKU defect should be fixed by row semantics, not by product-level exception list.
- Preferred implementation shape:
  1. Keep product-level aggregate for non-reward monetary distribution where safe.
  2. Do not propagate `reward_product` `IsProductPromotion=true` to normal rows during detail snapshot distribution.
  3. Ensure reward rows are created/read from `item_type=2` path and carry SO/final true, PO false.
- Final sync should happen only for no-proforma scope. If exact proforma field location differs, implementer must inspect query/model and use safe SQL predicate.
- No material open question blocks implementation.
