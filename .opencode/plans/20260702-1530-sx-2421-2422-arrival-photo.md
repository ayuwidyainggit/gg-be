# Plan — SX-2421 & SX-2422: Display Arrival Photo in Outlet Details (BE)

> Plan ini adalah source of truth untuk implementasi BE di service `pjp`. Setelah plan
> disetujui, eksekusi source edit dilakukan oleh lane `@orchestrator` + worker
> (`@fixer`/`@backend`) — planner `@artifact-planner` **tidak** mengedit source.

## Goal

Tambahkan field `file_url` di response `GET /scylla-pjp/v1/live-monitoring-distributor`
(SX-2421) dan `GET /scylla-pjp/v1/live-monitoring-principal` (SX-2422). Source of truth:
`mobile.visits.file_url`, dengan pemilihan visit terbaru per `(cust_id, emp_code, outlet_code, date)`
mengikuti pola yang sudah dipakai di distributor.

## Non-goals

- Tidak menambah endpoint baru.
- Tidak mengubah asal `arrive_longitude`/`arrive_latitude` (tetap dari
  `pjp_principles.outlet_visit_list ovl` untuk principal dan dari `mobile.visits` untuk
  distributor).
- Tidak mengubah pagination, sorting, atau shape response.
- Tidak menambah dependency baru.
- Tidak rename field existing.

## Scope (first slice)

Hanya 2 endpoint live monitoring. Modul: `pjp`. Sub-modul yang disentuh:

- `pjp/model/live_monitoring.go` — struct rows & LatestVisitCoordinateRow
- `pjp/data/response/live_monitoring_response.go` — DTO destination_data
- `pjp/repository/live_monitoring/get_distributor_repository.go`
- `pjp/repository/live_monitoring/get_principal_repository.go`
- `pjp/repository/live_monitoring/get_principal_extra_call_repository.go`
- `pjp/service/live_monitoring/get_distributor_service.go`
- `pjp/service/live_monitoring/get_principal_service.go`
- Test yang relevan: `get_distributor_service_test.go`, `get_principal_service_test.go`

## Requirements

1. **R1 — distributor endpoint return `file_url`.** Level `destination_data[]` di
   `data[].pjp_data[].{route_data, extra_call_data}[]`. Field: `file_url`.
2. **R2 — principal endpoint return `file_url`.** Sama level dengan R1.
3. **R3 — source of truth.** Nilai `file_url` berasal dari `mobile.visits.file_url` untuk kedua
   endpoint.
4. **R4 — visit row selection.** Pilih row dengan `ROW_NUMBER() OVER (PARTITION BY
   cust_id, emp_code, outlet_code ORDER BY created_at DESC) = 1` agar konsisten dengan pola
   distributor (`enrichDistributorRowsWithLatestVisits`).
5. **R5 — null safety.** Jika tidak ada visit / `file_url` kosong, field `file_url` di response
   JSON = `null` (mengikuti pola pointer `*string`). Tidak panic / tidak error.
6. **R6 — backward compatible.** Tidak ada field existing yang dihapus/rename. Tambahan
   hanya field baru.
7. **R7 — principal dual-branch.** Update baik `GetPrincipalMonitoring` (regular) maupun
   `GetPrincipalExtraCalls` (extra call) sehingga `file_url` tampil untuk semua destination di
   extra_call_data juga.
8. **R8 — validasi.** `rtk go build ./...` & `rtk go test ./service/live_monitoring/... ./repository/live_monitoring/... ./model/... ./data/response/...` lulus tanpa regression.

## Acceptance Criteria

- AC1. `GetDistributorMonitoring` mengembalikan row dengan `FileURL` yang sudah ter-enrich
  dari `mobile.visits` (latest per hari).
- AC2. `enrichDistributorRowsWithLatestVisits` melakukan assign `FileURL` seperti
  `ArriveLongitude/ArriveLatitude`.
- AC3. Response JSON distributor memuat `"file_url": <string|null>` di
  `destination_data[]`, level sama dengan `arrive_longitude/arrive_latitude`.
- AC4. `GetPrincipalMonitoring` (regular) & `GetPrincipalExtraCalls` mengembalikan
  `FileURL` di row-nya. Source tetap `mobile.visits.file_url` (bukan `ovl.file_url`).
- AC5. `transformPrincipalRows` melakukan assign `FileURL` ke
  `LiveMonitoringDestinationData`.
- AC6. Response JSON principal memuat `"file_url": <string|null>` di
  `destination_data[]` (untuk `route_data` dan `extra_call_data`).
- AC7. Field `arrive_longitude/arrive_latitude` existing **tidak** berubah nilai/semantik.
- AC8. `rtk go test ./service/live_monitoring/...` lulus termasuk assertion baru untuk
  `FileURL` di test yang relevan.
- AC9. Build service `pjp` lulus tanpa error.

## Existing Patterns / Reuse

- Pola **"latest visit enrichment"** sudah ada di distributor:
  `GetDistributorLatestVisitCoordinates` (`pjp/repository/live_monitoring/get_distributor_repository.go:181-223`)
  menggunakan subquery + `ROW_NUMBER() OVER (...)` lalu di-wrap dengan `row_number = 1`.
  Service lalu assign via map `buildDistributorVisitCoordinateKey(custID, empCode, outletCode)`.
- Pola **window function select** sudah ada di principal:
  `GetDistributorAttendance` (check-in/out) menggunakan `ROW_NUMBER() OVER (PARTITION BY
  me.emp_id ORDER BY a.created_at ...)`. Pola ini bisa dipakai ulang untuk join
  `mobile.visits` di principal.
- Pola **date range builder** sudah ada:
  `buildLiveMonitoringDayRange(date)` di `get_distributor_repository.go:35-44`. Sama
  akan dipakai untuk principal join.
- Pola **null-safe string**: gunakan `*string` (sama dengan `DestinationAddress *string`-like,
  lihat `response/live_monitoring_response.go:50-67`).

## Constraints

- **Shortest working diff.** Tidak ada abstraction baru; tidak ada helper baru kecuali
  reuse `buildLiveMonitoringDayRange`.
- Tidak rename field existing.
- Tidak ubah query builder besar; tambah kolom/join seminimal mungkin.
- Tidak tambah dependency baru.
- `FileURL` di JSON ditulis tanpa `omitempty` agar FE bisa membedakan `null` vs
  hilang; `*string` (nil) → `null`. Tetap konsisten dengan pattern `arrive_longitude`
  (yang bertipe `float64` non-pointer dan emit `0` untuk null). Karena field adalah
  string, defaultnya tetap `*string` nil → `null` di JSON Go. **Keputusan:** pakai
  `*string` (nil ⇒ `null`) — lebih jujur untuk FE yang minta tampil/sembunyikan foto.
- Cek schema `mobile.visits.file_url` sudah ada. **Confirmed via `pjp/data/request/visit_request.go:33,58`
  & `pjp/service/visit_service.go:99`** — endpoint mobile visit sudah mengirim/menyimpan
  `file_url`.

## Risks

- **R-1 (mismatch doc vs code principal).** Doc `monitoring_activity_be_doc.txt` section 5
  menulis principal query yang join `mobile.visits` untuk arrive_longitude; realitanya code
  principal sekarang ambil dari `ovl`. Plan ini **tidak** menyentuh `arrive_longitude`/
  `arrive_latitude` asal; hanya menambahkan `file_url` lewat join baru. Risiko rendah.
- **R-2 (window function performance).** Tambah window function + LEFT JOIN
  `mobile.visits` di principal regular + extra-call menambah beban query. Mitigasi:
  filter `mv.created_at` ke `visitStartAt/visitEndAt` (sudah pola distributor). Partisi
  pada `(me.emp_id, d.destination_code)`.
- **R-3 (multiple visits per outlet per hari).** Dipilih row_number=1 (terbaru) sesuai
  jawaban user & konsistensi distributor. Risiko FE perlu spesifik visit "pertama" — bisa
  diubah ke ASC di iterasi berikut dengan diff minimal (cuma ubah `ORDER BY`).
- **R-4 (Citus sharding).** `mobile.visits` dipakai oleh query Citus-friendly di
  distributor (lihat comment distributor repo line 10-15). Untuk principal, perlu cek
  sharding key `mobile.visits` adalah `cust_id` — sudah cocok karena `pjp_principles`
  juga sharded by `cust_id`. Partisi window function akan per `cust_id`.

## Decisions / Assumptions

- **D1.** Source `file_url` = `mobile.visits.file_url` (jawaban user).
- **D2.** Row selection = ROW_NUMBER=1 ORDER BY `created_at DESC` (jawaban user, konsisten
  dengan distributor).
- **D3.** Tipe `FileURL *string` di struct (nil = `null` di JSON). Field `json:"file_url"`
  tanpa `omitempty`.
- **D4.** Field `file_url` ditambahkan di level `destination_data[]` (sama level dengan
  `arrive_longitude`/`arrive_latitude`) — sesuai permintaan Jira.
- **D5.** Untuk principal, kita **tidak** mengubah query existing; kita tambahkan join
  `mobile.visits` dengan partisi yang benar dan pilih `v.file_url` via `MAX(...)` atau
  `ROW_NUMBER() = 1`. Pilih **pendekatan ROW_NUMBER() = 1 (subquery/CTE)** untuk
  konsistensi.
- **D6.** Service `pjp` (bukan `pjp-principle`) adalah satu-satunya handler live monitoring
  — `grep` di `pjp-principle` nol hasil. Diff hanya di service `pjp`.
