# Plan SX-2234 — Sales Trend `total_gross_sale` dan `net_sales`

## Goal

Ubah kalkulasi backend untuk:

```http
GET /sales/v1/reports/secondary-sales/trend-sales?year=2026&cust_id=C260020001
```

Target utama:

- `total_gross_sale` pakai finalized order detail dikurangi return detail.
- `net_sales` pakai formula order/return gross-discount + delta PPN.
- Response tetap 12 bulan, schema tetap sama.

## Non-goals

- Jangan ubah response wrapper `message`, `data`, `request_id`.
- Jangan tambah public filter baru untuk `outlet_ids`, `salesman_ids`, `product_ids` kecuali DTO sudah mendukung; saat discovery trend endpoint belum expose filter itu.
- Jangan ubah auth middleware atau token handling.
- Jangan edit `pjp-sales` kecuali owner deployment bilang module itu juga melayani endpoint ini.
- Jangan commit Bearer token, `.env`, dump DB, atau data sensitif.

## Scope

Target module: `sales`.

Target perubahan:

- `sales/repository/report_repository.go`
  - Ganti query `SecondarySalesReportTrendSales` dari `report.fact_orders` ke source tables `sls."order"`, `sls.order_detail`, `sls.return_det`, `sls."return"`.
  - Tambah helper query builder bila perlu agar test bisa assert SQL lebih mudah.
- `sales/repository/report_repository_test.go`
  - Tambah regression test dry-run untuk SQL trend baru.
- Opsional jika test butuh mapping coverage:
  - `sales/service/report_service_test.go` untuk 12 rows mapping bila repo mock mengembalikan 12 rows.

## Requirements

- `year` tetap required sesuai `sales/entity/report.go:234-238`.
- `cust_id` query/body tetap didukung.
- Multi-cust query tetap didukung: `?cust_id=CHILD1,CHILD2&cust_id=CHILD3`.
- Missing `cust_id` tetap fallback ke auth cust via service.
- Unauthorized cust tetap rejected via `ErrUnauthorizedCustID` dan controller 403.
- Month rows wajib `1..12`, ascending.
- Empty month values wajib zero.
- Return-only month boleh negatif; jangan clamp.
- SQL pakai parameter binding, bukan string interpolation.
- Date filter wajib half-open:
  - `dateFrom = time.Date(year, time.January, 1, 0, 0, 0, 0, time.UTC)`
  - `dateTo = dateFrom.AddDate(1, 0, 0)`
- Order side filter pakai `o.invoice_date >= ? AND o.invoice_date < ?`.
- Return side filter pakai `r.return_date >= ? AND r.return_date < ?` sesuai SX-2234 prompt.

## Acceptance Criteria

- `GET /sales/v1/reports/secondary-sales/trend-sales?year=2026&cust_id=C260020001` tetap HTTP 200 untuk authorized cust.
- `data.length == 12`.
- `data[0].month == 1`, `data[11].month == 12`.
- `total_gross_sale = order_gross - return_gross`.
- `net_sales = ((order_gross - order_discount) - (return_gross - return_discount)) + (order_ppn - return_ppn)`.
- `total_discount_promo` field tetap ada dan numeric.
- JSON field names unchanged.
- Existing controller/service tests for `cust_id` scope tetap lulus.
- Repository dry-run test membuktikan no `report.fact_orders` in trend query.
- Tidak ada real Bearer token di code/test/log/evidence.

## Implementation Steps

1. Baseline runtime check if local/staging auth available:
   - call endpoint with sanitized token only in local terminal, never store token.
   - record month count and April/empty-month values in task evidence.
2. Add repository Red test in `sales/repository/report_repository_test.go`.
3. Update `SecondarySalesReportTrendSales` in `sales/repository/report_repository.go`:
   - compute `dateFrom`, `dateTo`.
   - use months CTE.
   - use `order_summary` grouped by `EXTRACT(MONTH FROM o.invoice_date)::INTEGER`.
   - use `return_summary` grouped by `EXTRACT(MONTH FROM r.return_date)::INTEGER`.
   - bind `custIDs`, `dateFrom`, `dateTo`, `custIDs`, `dateFrom`, `dateTo`.
4. Gross formula:

```sql
SUM(
  (COALESCE(od.qty1_final, 0) * COALESCE(od.sell_price_final1, 0)) +
  (COALESCE(od.qty2_final, 0) * COALESCE(od.sell_price_final2, 0)) +
  (COALESCE(od.qty3_final, 0) * COALESCE(od.sell_price_final3, 0))
)
```

5. Discount formula for order:

```sql
SUM(
  COALESCE(od.disc_value_final, 0) +
  COALESCE(od.promo_final1, 0) + COALESCE(od.promo_final2, 0) +
  COALESCE(od.promo_final3, 0) + COALESCE(od.promo_final4, 0) +
  COALESCE(od.promo_final5, 0)
)
```

6. Return formulas:

```sql
SUM(
  (COALESCE(rd.qty1, 0) * COALESCE(rd.sell_price1, 0)) +
  (COALESCE(rd.qty2, 0) * COALESCE(rd.sell_price2, 0)) +
  (COALESCE(rd.qty3, 0) * COALESCE(rd.sell_price3, 0))
)
```

```sql
SUM(COALESCE(rd.disc_value, 0) + COALESCE(rd.promo_value, 0))
```

7. Final select aliases:

```sql
m.month AS month,
COALESCE(os.gross_sales, 0) - COALESCE(rs.gross_sales, 0) AS total_gross_sale,
COALESCE(os.discount_promo, 0) + COALESCE(rs.discount_promo, 0) AS total_discount_promo,
((COALESCE(os.gross_sales, 0) - COALESCE(os.discount_promo, 0)) -
 (COALESCE(rs.gross_sales, 0) - COALESCE(rs.discount_promo, 0))) +
(COALESCE(os.ppn, 0) - COALESCE(rs.ppn, 0)) AS net_sales
```

8. Run targeted tests.
9. Run full `rtk go test ./...` from `sales/` if targeted tests pass.
10. If runtime available, compare API output to direct SQL for one data month and one empty month. Store sanitized evidence.
