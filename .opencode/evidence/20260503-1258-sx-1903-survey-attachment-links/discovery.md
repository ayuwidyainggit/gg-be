# Discovery — SX-1903 Survey Attachment Links

## Files inspected

- `master/service/survey_report_service.go`
  - `Export` creates `DownloadSurveyReport-*`, fetches rows via `FindExportRows`, generates Excel, and stores base64 into download history via `ReportListRepository`.
  - `generateSurveyReportExcel` writes headers `Attachment 1`, `Attachment 2`, `Attachment 3` and writes `row.Attachment1/2/3` directly to cells.
- `master/repository/survey_report_repository.go`
  - `FindExportRows` selects attachment columns from `mst.survey_answer_files.file_key` using three subqueries with offsets 0..2.
  - Current attachment query is tenant-scoped with `saf.cust_id = sa.cust_id` and stable ordered by `saf.survey_answer_files`.
- `master/model/survey_report.go`
  - `SurveyReportExportRow` stores `Attachment1`, `Attachment2`, `Attachment3` as strings mapped from `attachment_1/2/3`.
- `master/adapter/obs-huawei.go`
  - Existing upload path builds public file URL as `https://<bucket>.<endpoint>/<key>` using `FileBaseUrl`.
  - Upload uses `obs.AclPublicRead`, so existing storage convention appears public-readable URL.
- `master/service/files.go`
  - File upload service delegates to `ObsAdapter.UploadFile` and returns the URL.
- `master/model/files.go`
  - `Upload.GenerateFileName` generates object key as `<folder>/<xid><extension>`.
- `master/main.go`
  - `obsAdapter` is initialized from `OBS_HUAWEI_AK`, `OBS_HUAWEI_SK`, `OBS_HUAWEI_ENDPOINT`, `OBS_HUAWEI_BUCKET`.
  - `surveyReportService` currently only receives `surveyReportRepository` and `reportListRepository`; it does not receive storage/URL resolver.
- `master/service/survey_report_service_test.go`
  - Existing test only asserts returned Excel is base64 xlsx bytes and currently uses raw `Attachment1: "display_1.jpg"`.
- `master/repository/survey_report_repository_test.go`
  - Existing tests validate query scoping/date behavior, not export attachment subquery behavior.
- `mobile/service/survey.go`, `mobile/repository/survey.go`, `mobile/model/survey.go`
  - Survey submit stores answer files in `mst.survey_answer_files`; mobile service currently appends `SurveyAnswerFile{FileName: f.ClientFileName}` in submit flow.
  - Submitted survey read fetches files by detail id and maps them back to question response.

## Project patterns found

- Service layer owns orchestration and mapping; repository should stay focused on DB query.
- Report export is synchronous in code path but stores result into `report_list`/Download History as base64 xlsx with status `READY`.
- Excel generation uses `github.com/xuri/excelize/v2`.
- Existing file upload in master returns public OBS URL generated by adapter using configured bucket and endpoint, not hardcoded in service.
- Root compose required check was run: `rtk docker compose -f docker-compose.yml ps`; `master`, `system`, and `redis` were up.

## Reuse candidates

- Reuse `ObsAdapterImpl.FileBaseUrl` URL convention rather than formatting domain in SQL or survey service.
- Extend OBS adapter with a narrow URL resolver method for object keys, then inject a narrow resolver interface into `SurveyReportService`.
- Reuse existing `generateSurveyReportExcel` style and add small pre-export row enrichment or pass resolver into Excel generation.
- Reuse existing `survey_report_service_test.go` for workbook assertions; decode generated xlsx and inspect cells with Excelize.
- Reuse existing `survey_report_repository_test.go` pattern for string assertions on generated query constraints.

## Constraints

- Do not hardcode credentials, bucket, endpoint, domain, or admin/Jira credentials.
- Preserve tenant scoping in attachment query: `saf.cust_id = sa.cust_id`.
- Preserve report headers, data order, Download History behavior, non-attachment fields, and existing filters.
- Existing root note says commands should use `rtk`; global OpenCode note says not to prefix with `rtk`. For this repo-specific planning evidence, the mandatory repo check was run with `rtk` because project `AGENTS.md` explicitly requires it.
- Source/test implementation edits must be done by implementation agent, not this planner.

## Risks

- Existing OBS upload uses public-read objects. Exporting full public URLs makes attachments easier to share externally; if survey attachments are sensitive/PII, private authorized download endpoint or signed/proxy flow may be needed as larger scope.
- Signed URL embedded into a stored Excel can expire before user downloads from Download History, so signed URLs are risky unless generated at click/download time through an authorized backend endpoint.
- If `file_key` can contain full external URLs, blindly preserving them would create phishing/open-link risk. Resolver should validate configured OBS host or treat values as object keys.
- If `mst.survey_answer_files` has soft-delete semantics, current export and `FindAnswerFiles` do not show `is_del` filtering; confirm schema before adding filter.
- Excel hyperlink support is possible through Excelize, but full URL text cells are lower risk and satisfy acceptable expected behavior unless product specifically requires hyperlink display text.
